## Integration approaches

There are two ways AI agents interact with TaskTrace:

| Approach | When to use |
|---|---|
| **MCP tools** | Agent is running inside Claude Code, Cursor, or another MCP-capable client |
| **CLI + JSON** | Agent uses shell execution (e.g. LangChain tools, custom agents) |

---

## MCP approach (recommended)

Configure the MCP server per the [MCP Server guide](/docs/mcp) and [Claude Code guide](/docs/guides/claude-code).

The agent calls tools directly — no shell execution, no JSON parsing:

```
worklog_status()                   → understand current state
worklog_add(...)                   → log a work event
worklog_consolidate()              → group events
worklog_attach_commit(sha="HEAD")  → link commits
```

## CLI + JSON approach

For agents that execute shell commands, every `tt` command supports `--json`:

<Tabs items={["Python", "TypeScript", "Shell"]}>
<Tab value="Python">

```python
import subprocess, json

def tt(args: list[str], cwd: str) -> dict:
    """Run a tt command and return parsed JSON output."""
    result = subprocess.run(
        ["tt", *args, "--json"],
        capture_output=True, text=True, cwd=cwd
    )
    result.check_returncode()
    return json.loads(result.stdout)

# Usage
status = tt(["status"], cwd="/path/to/project")
event  = tt(["add", "-d", "fixed race condition", "-k", "bugfix"], cwd="/path/to/project")
consolidated = tt(["consolidate"], cwd="/path/to/project")
```

</Tab>
<Tab value="TypeScript">

```typescript
import { execFile } from "node:child_process";
import { promisify } from "node:util";

const run = promisify(execFile);

// Always use execFile (not exec) — no shell injection risk
async function tt(args: string[], cwd: string): Promise<unknown> {
  const { stdout } = await run("tt", [...args, "--json"], { cwd });
  return JSON.parse(stdout);
}

// Typed helpers
const ttStatus   = (cwd: string) => tt(["status"], cwd);
const ttLog      = (cwd: string, n = 10) => tt(["log", "-n", String(n)], cwd);
const ttAdd      = (cwd: string, desc: string, kind = "implementation") =>
  tt(["add", "-d", desc, "-k", kind, "--source", "ai"], cwd);
const ttConsolidate = (cwd: string) => tt(["consolidate"], cwd);

// Example: session wrap-up
async function sessionWrapUp(projectDir: string) {
  await ttConsolidate(projectDir);
  const status = await ttStatus(projectDir) as { pendingEntries: number; totalEntries: number };
  return `Logged ${status.totalEntries} entries. ${status.pendingEntries} pending sync.`;
}
```

</Tab>
<Tab value="Shell">

```bash
#!/usr/bin/env bash
# session-end.sh — wrap up an AI coding session

PROJECT_DIR="${1:-.}"

# Consolidate events
RESULT=$(tt consolidate --json)
CREATED=$(echo "$RESULT" | jq '.createdEntries | length')

# Get final status
STATUS=$(tt status --json)
PENDING=$(echo "$STATUS" | jq '.pendingEntries')
BRANCH=$(echo "$STATUS" | jq -r '.currentBranch')

echo "Session complete on branch: $BRANCH"
echo "Created $CREATED entries. $PENDING pending sync."
echo ""
echo "Run 'tt plan' to review, then 'tt sync' to send to ClickUp."
```

## Recommended agent patterns

### Pattern 1 — Log as you go (best)

Log small events frequently rather than one big event at the end.

```
agent investigates issue  → worklog_add("investigated X, found Y", kind="investigation", durationMinutes=15)
agent implements fix      → worklog_add("fixed X by doing Z", kind="bugfix", durationMinutes=30)
agent writes tests        → worklog_add("added tests for X edge cases", kind="implementation", durationMinutes=20)
agent commits             → worklog_attach_commit()
session ends              → worklog_consolidate()
```

**Why:** Granular events produce better entry summaries after consolidation. One vague
event ("worked on auth") is less useful than three specific events.

### Pattern 2 — Session bookends

Simpler: one event at the start, one at the end, attach on commit.

```
session starts  → worklog_add("starting: fix auth timeout CU-abc123", kind="investigation")
... work ...
commit          → worklog_attach_commit()
session ends    → worklog_add("completed: fixed auth timeout with read-write lock", kind="bugfix", durationMinutes=60)
                → worklog_consolidate()
```

### Pattern 3 — Status-driven decisions

The agent reads `worklog_status` to decide what to prompt the user:

```
get worklog_status()

if pendingEvents > 5:
  → call worklog_consolidate()

if pendingEntries > 0:
  → tell user "You have pending entries. Run `tt sync` when ready."
```

## Entry improvement workflow

AI agents can improve entry summaries after consolidation via the `WorklogEngine` API:

```typescript
import { WorklogEngine, FileStorageProvider, loadConfig } from "@tasktrace/core";

const config = await loadConfig({ dir: projectDir });
const engine = new WorklogEngine({
  config,
  storage: new FileStorageProvider(config.storage.dir),
});

// Consolidate
const entries = await engine.consolidate();

// Improve each summary with your LLM
for (const entry of entries) {
  const betterSummary = await myLLM.summarize(entry.summary, entry.eventIds);
  await engine.updateEntry(entry.id, { summary: betterSummary });
}
```

## What agents should never do

- **Auto-sync** — always leave `tt plan` + `tt sync` to the human
- **Delete events** — the append-only log is the source of truth
- **Modify event timestamps** — events are immutable by design
- **Skip `source: "ai"`** — always identify AI-created events for auditability

## LLM-friendly context injection

If you're building a custom agent, inject current worklog context into the system prompt:

```bash
# Get context as JSON
STATUS=$(tt status --json)
RECENT=$(tt log -n 5 --json)

# Inject into your agent's context
CONTEXT="Current worklog state: $STATUS. Recent entries: $RECENT"
```

Or use the MCP resources directly:
- `tasktrace://config` — project configuration
- `tasktrace://worklogs/recent` — last 10 entries
